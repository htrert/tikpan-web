# ADR-0001: 多渠道模型网关架构

## 状态

已采纳，2026-05-13。

## 背景

项目需要把 ComfyUI 节点里已经验证过的模型参数迁移到网站。用户不再下载 ComfyUI，而是在网站里选择模型和参数。网站负责把请求转发给 Tikpan 或其他上游中转站，并完成充值、扣费、渠道路由、结果返回。

## 决策

采用“网站前台 + 管理后台 + 多渠道模型网关”的架构：

- 前台只展示中文友好的模型参数和上传控件。
- 后台配置模型字段、价格、供应商渠道和模型路由。
- 每个生成请求先扣费并写入余额流水，再调用上游。
- 上游失败自动退款，退款同样写入余额流水。
- 媒体结果优先保存到 OSS/CDN，不由应用服务器长期承载。
- 请求支持 `Idempotency-Key`，用于降低重复点击和网络重试造成的重复扣费风险。

## 选择理由

- 和 ComfyUI 节点思路一致，便于复用已有参数经验。
- 管理员可以调整渠道和价格，不需要每次改代码。
- 多渠道能降低单一中转站故障或涨价带来的业务风险。
- 余额流水让财务和客服有证据可查。
- OSS/CDN 让 4H4G + 30M 的起步服务器也能承载图片、音频、视频结果。

## 取舍

- 当前保留 SQLite，部署简单；用户量上来后迁移 PostgreSQL/MySQL。
- 当前先做同步生成请求；长视频、音乐等任务后续迁移到队列。
- 当前路由以优先级为主；后续再加入自动健康检查、权重和成本优化。
