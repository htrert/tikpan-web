# 新增模型与节点同步流程

## 1. 推荐更新方式

后续 Tikpan 或上游出了新模型，不建议直接在后台临时手填一堆字段作为唯一来源。推荐流程是：

1. 先在 ComfyUI 节点里验证参数是否能稳定跑通。
2. 把模型、端点、字段、下拉选项、默认值写入 `web_app/catalog/tikpan_nodes.py`。
3. 运行同步脚本，把 catalog 同步到网站数据库。
4. 在后台检查字段展示、价格、渠道路由。
5. 用 `/api/request-preview` 预览请求，再用小额任务实测。

这样做的好处是：换电脑、换服务器、重建数据库时，模型配置不会散落在人工后台操作里。

## 2. 运行同步

预览将同步多少内容：

```bash
cd web_app
python scripts/sync_tikpan_nodes.py --dry-run
```

真正同步：

```bash
python scripts/sync_tikpan_nodes.py
```

也可以在管理后台“关于”页面点击“同步本地 Tikpan 节点到网站”。

## 3. catalog 里要填什么

每个模型至少需要：

- `id`: 网站内部模型 ID，稳定后不要随便改。
- `node_class`: 对应 ComfyUI 节点类名，方便追溯。
- `category_key`: 网站分类。
- `name`: 前台展示名称。
- `provider`: 供应商说明。
- `description`: 给用户看的简介。
- `api_type`: 后端调用类型，成熟模型用专用 handler，新模型先用 `tikpan_proxy`。
- `endpoint`: 上游接口路径。
- `upstream_model`: 上游真实模型名。
- `pricing`: 默认计费规则。
- `fields`: 前台表单字段，尽量用下拉选项，不让用户手填难懂参数。

## 4. 什么时候需要写专用 handler

`tikpan_proxy` 适合新模型快速接入和测试，但下面情况建议写专用 handler：

- 需要 multipart 上传文件。
- 返回值格式复杂，需要轮询任务。
- 要把 base64 图片保存到 OSS/CDN。
- 要对视频、音频做下载和缓存。
- 需要非常严格的错误分类和退款逻辑。

成熟商业模型应逐步从 `tikpan_proxy` 升级为专用 handler。

## 5. 上线前检查

每次新增模型后至少跑：

```bash
python -B -m py_compile web_app/catalog/tikpan_nodes.py web_app/scripts/sync_tikpan_nodes.py web_app/backend/handlers.py
python scripts/sync_tikpan_nodes.py --dry-run
python scripts/deploy_check.py
```

然后后台检查：

- 模型是否出现在正确分类。
- 字段顺序是否符合用户使用习惯。
- 下拉选项是否比手填更易懂。
- 价格是否覆盖成本并留有利润。
- 路由是否绑定 `tikpan-default` 或其他正确渠道。
- 实测失败时是否退款并写入余额流水。
