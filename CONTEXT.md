# Tikpan Web App Context

## 产品定位

Tikpan Web App 是一个面向中文用户的 AI 模型聚合服务入口。用户在网站里选择模型、填写参数、上传参考图/音频/视频，网站后端把这些参数转换成上游 API 请求，按配置的渠道路由到 Tikpan 或其他中转站，再把结果返回给用户。

目标不是让用户学习 ComfyUI，而是把已经验证过的 ComfyUI 节点参数产品化为网站表单、下拉框、上传控件和可视化后台配置。

## 核心链路

用户 -> 网站前台 -> 余额/权限校验 -> 请求幂等键 -> 模型参数规范化 -> 渠道路由 -> 上游 API -> OSS/CDN 存储 -> 返回结果 -> 生成记录与余额流水

## 商业化原则

- 用户看到的是中文友好的参数名和下拉选项，不直接暴露难懂的官方字段。
- 管理员在后台配置模型、字段、定价、供应商渠道和路由，尽量不改代码。
- 每次扣费必须能追溯到请求、订单或代理动作。
- 上游失败、超时、格式错误时要自动退款并记录原因。
- 大文件结果不走应用服务器直出，应该上传 OSS/R2/S3，再通过 CDN 访问。
- `.env`、数据库、输出文件、恢复记录和 API key 不进 Git。

## 当前技术栈

- Flask + SQLite，适合启动期和单机部署。
- Docker + Gunicorn + Nginx 反代用于生产部署。
- 后续用户量上来后，优先把 SQLite 迁移到 PostgreSQL/MySQL，再引入 Redis 队列。

## 重要目录

- `api/`: 前台 API，例如登录、充值、生成、代理。
- `backend/`: 管理后台、模型配置读取、上游处理器。
- `models/`: SQLite 表结构和数据访问函数。
- `templates/`: 前台和后台页面。
- `scripts/`: 配置迁移、部署检查、数据库初始化脚本。
- `docs/`: 部署、迁移、架构和运维手册。

## 改动成功标准

- 代码能通过 `py_compile`。
- `git diff --check` 无空白错误。
- 新增模型或字段尽量有下拉选项、默认值和中文说明。
- 涉及扣费的功能必须写 `balance_ledger`。
- 涉及上游请求的功能必须考虑超时、失败退款、幂等和日志。
